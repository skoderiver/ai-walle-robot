# 自主导航与大模型可行性调研

![自主能力栈](./assets/autonomy_stack.svg)

## 1. 先说结论

你的目标可以拆成三部分：

- `自然对话`：可行，而且现在已经比较成熟
- `室内自主移动到指定地点`：可行，但需要完整导航栈，不再是普通玩具机器人级别
- `自己看懂真实世界、上楼梯到二楼`：理论可做，但对个人 DIY 的瓦力项目来说是高风险目标，不能默认当成 V1

最重要的判断是：

`大模型可以做“理解任务、规划步骤、调用工具、对话回复”，但不能替代机器人底盘、定位、避障、楼梯运动机构和安全控制。`

## 2. 你说的能力拆开后分别是什么

### 2.1 和你对话

这部分相对容易：

- 麦克风采集语音
- ASR 语音转文本
- LLM 理解你的指令
- TTS 把回复说出来

### 2.2 听懂“去二楼”

这不是一句话理解的问题，而是任务规划问题。它至少要会：

- 识别“二楼”对应的目标地点
- 查询地图中有没有该地点
- 判断自己当前所处楼层
- 判断是否具备跨楼层运动能力
- 如果没有，必须明确拒绝或请求人工协助

### 2.3 真正走到二楼

这一步最难，因为它同时要求：

- 可靠的室内定位
- 真实世界避障
- 足够稳定的底盘
- 能跨越楼梯这种 3D 地形变化
- 发生意外时能及时停下

## 3. 自主平地导航是可行的

如果你把目标改成：

- “去客厅”
- “去门口”
- “去充电站”
- “去一楼会议区”

这类平地目标是现实可做的。

比较成熟的工程路线是：

- 激光雷达或深度相机建图
- IMU + 轮速里程计辅助定位
- ROS2 + Nav2 做 2D 导航
- 高层代理把自然语言地点翻译成导航目标点

Nav2 官方文档说明它是 ROS 导航栈的后继者，用于移动机器人在复杂环境中完成导航任务；同时它的环境表示核心是 `2D costmap`。这意味着它非常适合平地室内导航，但“上楼梯”不是它默认擅长的问题。这是我基于官方文档做的工程推断。来源：

- [Nav2 Overview](https://docs.nav2.org/)
- [Nav2 Concepts](https://docs.nav2.org/concepts/index.html)

## 4. “自主上楼”为什么难

楼梯问题不是普通导航问题，而是 `机动能力` 问题。

如果底盘本身没有楼梯通过能力，再强的大模型也没法让它上楼。

### 4.1 普通瓦力底盘的问题

普通 DIY 瓦力项目常见的是：

- 双轮差速
- 普通小履带
- 轻量装饰外壳

这类结构适合：

- 平地
- 小门槛
- 缓坡

但不适合：

- 标准楼梯
- 不规则踏步
- 边缘缺口
- 重心频繁前后转移的地形

### 4.2 能上楼的机器人通常长什么样

能上楼的商业/工程机器人，通常会有至少一项：

- 足式/四足平台
- 大扭矩重型履带
- 翻转臂/辅臂
- 专门针对楼梯设计的重心和控制策略

例如 Boston Dynamics 公开资料显示，Spot 的楼梯通过能力依赖整机感知、控制与平台能力结合，而不是单纯“看见楼梯就去走”。来源：

- [Spot 2.0: Autonomy & Mobility](https://bostondynamics.com/blog/spot-2-0-autonomy-mobility/)
- [Spot 产品页](https://bostondynamics.com/products/spot/)

另一个更接近履带路线的制造商资料显示，带 `flipper arm/stabilizer arm` 的履带机器人可以爬家用楼梯。这说明如果你坚持“瓦力风格 + 上楼”，机械路线大概率要改成带翻转辅臂的特种履带，而不是普通展陈履带。来源：

- [SuperDroid LT2-F 数据表 PDF](https://www.superdroidrobots.com/product_info/SDR_LT2-F.pdf)

这是我基于公开资料做的工程推断，不是说你的项目必须完全照搬这些平台。

## 5. 大模型在机器人里应该负责什么

### 适合让 LLM 做的事

- 理解自然语言
- 把“去二楼找我”翻译成结构化任务
- 询问缺失信息
- 调用工具
- 结合地图、传感器状态和任务结果做高层决策
- 和人自然对话

### 不适合直接交给 LLM 的事

- 毫秒级电机控制
- 急停
- 姿态稳定
- 编码器闭环
- 楼梯踏步级的实时平衡

LLM 应该是 `任务大脑`，不是 `电机控制器`。

## 6. OpenAI / GPT 路线是否可行

可行，而且比较适合做高层代理。

根据 OpenAI 当前官方文档：

- 官方建议开发者优先迁移到 `Responses API`
- `Conversations` 和 `Responses` 更适合多轮会话与工具调用
- 自定义工具调用可以让模型调用导航、查询地图、读取电量等你自己的工具
- `gpt-realtime` 适合低延迟语音交互

来源：

- [Responses API 指南](https://platform.openai.com/docs/guides/responses-vs-chat-completions)
- [Conversations API Reference](https://developers.openai.com/api/reference/resources/conversations)
- [gpt-realtime 模型页](https://developers.openai.com/api/docs/models/gpt-realtime)

对于语音链路，OpenAI 当前也有可直接用的语音模型：

- `gpt-4o-transcribe` / `gpt-4o-mini-transcribe` 做语音识别
- `gpt-4o-mini-tts` 或 `tts-1` 做语音播报

来源：

- [gpt-4o-transcribe](https://developers.openai.com/api/docs/models/gpt-4o-transcribe)
- [gpt-4o-mini-transcribe](https://developers.openai.com/api/docs/models/gpt-4o-mini-transcribe)
- [gpt-4o-mini-tts](https://developers.openai.com/api/docs/models/gpt-4o-mini-tts)
- [tts-1](https://developers.openai.com/api/docs/models/tts-1)

## 7. 推荐你采用的能力分层

### 层 1：安全与执行层

- ESP32 / STM32
- 电机、舵机、急停、欠压、限位、失联停车

### 层 2：导航与感知层

- Jetson Orin Nano 或更高
- ROS2
- Nav2
- 激光雷达或深度相机
- IMU + 编码器

### 层 3：任务代理层

- Python 服务
- OpenAI Responses API
- 工具调用：`navigate_to(location)`、`get_robot_state()`、`speak(text)`、`ask_for_help()`

### 层 4：交互层

- 麦克风
- 扬声器
- 屏幕/眼睛动画
- Unity 仿真与状态可视化

## 8. 一个现实可落地的路线图

### V1：不做人工驾驶，只做“监督式自主”

- 支持语音输入
- 能回答问题
- 能去预设平地点位
- 人可随时急停

### V2：半自主多房间导航

- 自动建图
- 语义地点管理
- 避障
- 失败后能请求帮助

### V3：跨楼层方案

只有在你明确选择特殊机械平台时才建议启动。

可选路线：

- `路线 A`：接入电梯，不爬楼梯
- `路线 B`：改为特种履带 + 翻转臂楼梯平台，外面再做瓦力外观包覆
- `路线 C`：放弃瓦力外形，改用足式平台

如果你真的想说“去二楼”，我最推荐的是：

`先做会坐电梯的跨楼层机器人，而不是先做会爬楼梯的瓦力。`

## 9. 我给你的现实建议

### 如果你想尽快成功

把目标改成：

- 语音对话
- 平地自主移动
- 能去房间和充电站
- 会解释自己正在做什么

### 如果你坚持“上二楼”

需要接受这些变化：

- 外形优先级会下降
- 机械和安全成本会飙升
- 调试周期会显著变长
- 预算会从展示机器人进入特种移动机器人区间

## 10. 一句话结论

`会说话 + 会看 + 会在平地自主去指定地点` 是现实可做的。

`瓦力外形 + 自主爬楼梯去二楼` 对个人 DIY 来说不是单纯的软件升级，而是机械平台路线变化，应该被当成高风险独立项目。
