# 技术方案选型

![系统架构示意](./assets/system_architecture.svg)

## 1. 总体架构建议

推荐采用“两层控制架构”：

- `底层实时控制`：ESP32 / STM32
- `高层计算与交互`：树莓派 5 或 NVIDIA Jetson Orin Nano

这样做的原因：

- 电机、舵机、编码器、急停这些任务要求实时性，MCU 更稳
- 视觉、语音、ROS2、AI 推理更适合 Linux 平台
- 高低层分离后，单个模块坏了更容易替换

## 2. 三档方案

### 方案 A：最低风险展陈版

- MCU：ESP32
- 高层：无，或仅手机/Web 遥控
- 优点：成本低、开发快、调试简单
- 缺点：扩展 AI 能力弱

### 方案 B：平衡版

- MCU：ESP32
- 高层：树莓派 5
- 优点：适合音视频控制、Web 面板、ROS2 入门
- 缺点：本地 AI 推理能力一般

### 方案 C：英伟达风格 AI 展示版

- MCU：ESP32
- 高层：Jetson Orin Nano Developer Kit
- 优点：适合做本地视觉、语音、ROS2、NVIDIA 技术栈展示
- 缺点：成本、功耗、散热、集成复杂度更高

## 3. 我对你的推荐

如果你明确想做“英伟达展览风格”，推荐从 `B 架构启动，按 C 架构预留接口`：

- 第一阶段先用 `ESP32 + 树莓派/笔记本 + Unity`
- 第二阶段把高层替换或升级为 `Jetson`

这样不会在硬件采购初期就被 AI 复杂度拖慢。

## 4. 通讯方案

### MCU 到高层

- 首选：`UART`
- 次选：`USB Serial`
- 如果模块较多：`CAN` 或 `RS485`

第一版不建议把核心控制链路完全押宝在 Wi-Fi 上。

### 高层到遥控端

- WebSocket
- HTTP API
- ROS2 Topic
- Unity 通过 TCP/UDP/ROS-TCP Connector 对接

## 5. 视觉与 AI 选型

### 摄像头

- USB 摄像头：开发最快
- CSI 摄像头：Jetson/树莓派 集成度更高
- 深度摄像头：后期升级，第一版可不买

### 语音

- 第一版：离线唤醒词或预设按钮触发
- 第二版：Whisper 类 ASR 或云端识别
- 第三版：本地小模型或云端大模型对话

## 6. 软件中间件

### 不上 ROS2 的版本

适合 V1。

- Unity 控制台 / Web 控制台
- Python/Node.js 高层服务
- 串口协议下发到底层

### 上 ROS2 的版本

适合 V2/V3。

- 底层通过 microcontroller 协议桥接
- 高层以 ROS2 节点组织摄像头、状态机、动作脚本
- Unity 通过 ROS-TCP Connector 或自定义桥接获取状态

## 7. 关键选型建议

### 底盘驱动电机

- 优先选带减速箱的直流电机
- 若要速度闭环，优先选带编码器版本

### 姿态关节

- 小负载部位可用数字舵机
- 头部和大扭矩部位尽量不要用廉价塑料齿舵机

### 供电

- 动力与逻辑供电分离
- 舵机供电单独稳压
- 所有地线共地

## 8. 推荐结论

### 你如果重视成功率

选：

- ESP32
- 树莓派 5 或现有 PC
- 双履带直流减速电机
- 数字舵机控制头部/手臂
- Unity 做上位机与调试界面

### 你如果重视“英伟达味道”

选：

- ESP32
- Jetson Orin Nano
- USB/CSI 摄像头
- ROS2 + Jetson 本地视觉
- Unity 做数字孪生与演示控制台

## 9. 选型图解

### 两层控制为什么重要

- 上层 Linux 板卡适合跑视觉、语音、状态机和日志
- 底层 MCU 适合处理电机、舵机、急停和心跳保护
- 分层后，就算高层进程卡顿，底层也能停车和保护

### 你的推荐组合

- `第一阶段`：ESP32 + 电脑/Unity + 串口
- `第二阶段`：ESP32 + 树莓派 5 + Unity
- `第三阶段`：ESP32 + Jetson Orin Nano + ROS2 + Unity
